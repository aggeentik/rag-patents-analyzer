# Patents Analyzer — Scaling Roadmap
### Demo → PoV → MVP → Production

**Prepared for:** Ferrosyn
**Date:** February 2026
**Solution:** Agentic RAG system for patent intelligence

---

## Executive Summary

This document outlines a phased scaling strategy for the Patents Analyzer, progressing from a validated demo prototype through Proof-of-Value and MVP to full production deployment. Two deployment scenarios are evaluated in parallel — **On-Premise** (air-gapped or VPN-connected IT landscape) and **AWS Cloud** — across four dimensions: technology stack, resiliency, scalability, and costs.

The current demo operates on 9 steel manufacturing patents with a single-user Streamlit interface. Each stage expands corpus size, user base, and system maturity while preserving the core hybrid retrieval architecture (BM25 + Semantic + Knowledge Graph + LLM).

## Stage 1: Demo → Proof-of-Value (4–6 weeks)

**Objective:** Validate solution fit with a focused patent corpus, integrate with Ferrosyn IT systems, and measure defined success metrics with a small pilot team.

### Scenario A — On-Premise

#### Technology Stack
- **Application:** Current Streamlit app deployed on a single Linux server (bare metal or VM); no architecture changes required.
- **LLM:** Ollama with `llama3.1:8b` running locally on the same server (8–16 GB VRAM GPU or CPU-only with acceptable latency), or Azure AI GPT-4.1 via outbound HTTPS if network policy permits.
- **Vector search:** FAISS (file-based) — unchanged from demo.
- **Knowledge Graph:** SQLite + NetworkX — unchanged from demo.
- **Storage:** Local filesystem for patent PDFs and processed indices.
- **Deployment:** Docker Compose single-host stack. No orchestration overhead.
- **Auth:** Shared credentials or basic HTTP auth via Streamlit. AD integration deferred to MVP.
- **Integration:** Manual PDF ingestion via CLI (`uv run python src/data_ingestion.py`).

#### Resiliency
- Single node — acceptable for a PoV with 3–5 users.
- Daily manual backup of `data/processed/` to network share.
- Restart script for automatic process recovery.
- No formal SLA required.

#### Scalability
- Not a primary concern; pilot team queries sequentially.
- Streamlit supports ~5 concurrent users before performance degrades — sufficient for PoV.

#### Costs
| Item | Estimate |
|---|---|
| Server hardware (if procured) | €5,000–12,000 one-time |
| GPU (if on-prem LLM required) | €3,000–8,000 (NVIDIA RTX 4090 or A4000) |
| Engineering (setup + onboarding) | €15,000–25,000 |
| Azure AI API costs (if used) | €200–500/month |
| **Total PoV** | **€20,000–45,000** |

### Scenario B — AWS Cloud

#### Technology Stack
- **Compute:** Single EC2 instance — `c5.2xlarge` (8 vCPU, 16 GB RAM) for CPU-only, or `g4dn.xlarge` (NVIDIA T4 GPU) if on-prem LLM is preferred over Bedrock.
- **LLM:** Amazon Bedrock (`anthropic.claude-3-haiku` or `meta.llama3-8b`) — pay-per-token, no infrastructure to manage.
- **Vector search:** FAISS on EFS (shared filesystem) — unchanged from demo.
- **Knowledge Graph:** SQLite on EFS — unchanged from demo.
- **Storage:** S3 bucket for raw patent PDFs; EFS for processed indices.
- **Networking:** VPC with private subnet; access via VPN or AWS Client VPN for Ferrosyn employees.
- **Deployment:** Docker on EC2 via `docker compose`. No orchestration.
- **Auth:** AWS Cognito user pool (basic) or IP allowlisting.

#### Resiliency
- Single AZ — acceptable for PoV.
- Automated daily EBS/EFS snapshots via AWS Backup.
- CloudWatch basic alarms (CPU, memory, disk).
- EC2 auto-recovery on instance failure.

#### Scalability
- Bedrock scales automatically — no action needed.
- EC2 is a single point; sufficient for 3–5 pilot users.

#### Costs
| Item | Estimate |
|---|---|
| EC2 `c5.2xlarge` (720 h/month) | ~€250/month |
| Amazon Bedrock (Claude Haiku, ~10K queries) | ~€50–150/month |
| S3 + EFS storage | ~€20–50/month |
| VPN + data transfer | ~€50/month |

## Stage 2: PoV → MVP (8–12 weeks)

**Objective:** Expand patent corpus to 100–1,000 documents, enable multi-department access (20–30 users), and harden retrieval and generation for domain-specific accuracy.

### Scenario A — On-Premise

#### Technology Stack
- **LLM:** Dedicated GPU server (`Llama 3.1 70B` or `Mistral Large` via vLLM inference server). Alternatively, maintain Azure AI connection via corporate proxy if network policy allows. vLLM provides OpenAI-compatible API — zero application code changes required.
- **Vector search:** Upgrade FAISS → **Qdrant** (self-hosted Docker container). Qdrant supports concurrent writes, persistence, filtering, and hybrid search. Provides REST + gRPC API compatible with current retrieval layer via a thin adapter.
- **Knowledge Graph:** Upgrade SQLite → **PostgreSQL** for concurrent multi-user access. NetworkX graph traversal layer remains unchanged; only the persistence backend changes.
- **Ingestion pipeline:** Introduce **Apache Airflow** (lightweight, Docker-hosted) for scheduled ingestion: watch `data/raw/` for new PDFs, trigger extraction, update indices, send completion notification.
- **Auth:** Active Directory / LDAP integration via Streamlit-Authenticator or a lightweight reverse proxy (Nginx + LDAP module). Role-based access: read-only vs. admin.
- **Storage:** NAS mount for patent PDF archive. Processed indices on local SSD.
- **Deployment:** **Docker Compose** extended, or migrate to **k3s** (lightweight Kubernetes) for rolling updates and service isolation.
- **Monitoring:** Prometheus + Grafana stack for query latency, LLM throughput, and index health.

#### Resiliency
- 2-node active/passive setup for the application layer (Nginx load balancer).
- Qdrant configured with replication factor 2.
- PostgreSQL with streaming replication to standby.
- Nightly automated backups to NAS with 30-day retention.
- Health checks and auto-restart via Docker/k3s.
- Target availability: **99% during business hours**.

#### Scalability
- vLLM inference server handles concurrent requests with batching — supports 10–20 simultaneous users.
- Qdrant scales vertically (add RAM/disk) and horizontally (add nodes) without downtime.
- Airflow handles burst ingestion by queuing tasks.

#### Costs
| Item | Estimate |
|---|---|
| GPU server (A100 40 GB or 2× A6000) | €30,000–60,000 one-time |
| Qdrant + PostgreSQL server | €8,000–15,000 one-time |
| NAS storage (10 TB) | €3,000–6,000 one-time |
| Software licences (Airflow OSS, Grafana OSS) | €0 (open source) |
| Engineering (refactoring + integration + hardening) | €60,000–90,000 |
| Operations (annual) | €20,000–35,000 |
| **Total MVP** | **€100,000–170,000** |

### Scenario B — AWS Cloud

#### Technology Stack
- **Compute:** Migrate from single EC2 → **ECS Fargate** (serverless containers) for the Streamlit app and ingestion workers. No servers to manage; scales to zero when idle.
- **LLM:** Amazon Bedrock with **on-demand pricing** (`claude-3-5-sonnet` for quality, `claude-3-haiku` for speed). No infrastructure.
- **Vector search:** **Amazon OpenSearch Service** (1 × `r6g.large` node to start) with k-NN vector search plugin. Drop-in replacement for FAISS via a thin adapter; supports filtering by patent ID, section, and date.
- **Knowledge Graph:** **Amazon Neptune** (serverless tier) for managed graph storage + NetworkX for in-process traversal. Alternatively, keep PostgreSQL + graph-in-Python approach on **RDS PostgreSQL** (cheaper, simpler).
- **Ingestion pipeline:** **S3 Event Notifications → SQS → ECS Task** for event-driven ingestion on new PDF upload. No cron jobs; new patents trigger processing automatically within minutes.
- **Auth:** **AWS Cognito** with SAML federation to Ferrosyn's IdP (Azure AD / ADFS). ALB authenticates users before traffic reaches the app.
- **Storage:** S3 for raw PDFs (versioned, lifecycle to Glacier after 2 years). EFS for processed index files shared across ECS tasks.
- **Deployment:** ECS Fargate with rolling deployments and CodePipeline for CI/CD.
- **Monitoring:** CloudWatch Dashboards + X-Ray tracing + SNS alerts.

#### Resiliency
- ECS Fargate: Multi-AZ by default, task auto-replacement on failure.
- OpenSearch: 2-AZ replication. Automated daily snapshots to S3.
- RDS/Aurora PostgreSQL: Multi-AZ standby, automated failover < 60 s.
- S3: 11 nines durability, cross-region replication optional.
- Target availability: **99.5%** (managed service SLAs).

#### Scalability
- ECS Fargate auto-scales app containers based on CPU/memory.
- Bedrock scales to millions of requests automatically.
- OpenSearch scales by adding nodes without downtime.
- S3 + SQS ingestion queue handles burst uploads gracefully.

#### Costs
| Item | Monthly Estimate |
|---|---|
| ECS Fargate (app + workers, ~40 h/day active) | €400–700 |
| Amazon Bedrock (Claude, ~50K queries/month) | €600–1,500 |
| OpenSearch Service (`r6g.large`, 1 node) | €200–350 |
| RDS PostgreSQL (`db.t4g.medium`, Multi-AZ) | €150–250 |
| S3 storage (1 TB patents + indices) | €25–50 |
| Cognito + ALB + VPN | €100–200 |
| CloudWatch + misc | €50–100 |
| **Running total** | **€1,500–3,150/month** |
| Engineering (refactoring + integration) | €50,000–80,000 one-time |

## Stage 3: MVP → Production

**Objective:** Enterprise integration with existing document management systems, continuous automated ingestion for newly filed patents, and Agentic AI workflows.

### Scenario A — On-Premise

#### Technology Stack
- **LLM inference:** Scale GPU cluster to 2–4 nodes. **vLLM** with tensor parallelism across GPUs. Model: `Llama 3.1 70B` or a fine-tuned domain-specific variant. LLM gateway (e.g., **LiteLLM Proxy**) for centralised rate limiting, logging, and model routing.
- **Vector search:** **Qdrant cluster** (3+ nodes, distributed mode) with sharding and replication. Supports 10M+ vectors.
- **Knowledge Graph:** **Neo4j Enterprise** for production-grade graph DB with Cypher query language, ACID transactions, clustering, and Enterprise security. Replaces SQLite/NetworkX for persistence while NetworkX is retained for in-memory traversal.
- **Ingestion pipeline:** **Apache Airflow** with dedicated worker nodes. DAGs triggered by: (1) filesystem watch on NAS/SharePoint, (2) scheduled EPO/USPTO API polling for newly published patents, (3) manual upload via UI. Patent processing SLA: < 2 hours from filing to searchable.
- **Document management integration:** Connectors for **SharePoint on-premise** (via Microsoft Graph API or SharePoint REST), **OpenText**, or **Documentum** — whichever Ferrosyn uses. Bi-directional: ingest from DMS, annotate with analysis results back to DMS.
- **Agentic AI:** **LangGraph** agent framework running on-prem. Three primary agents: (1) *Ingestion agent* — monitors, classifies, and routes new patents; (2) *Alert agent* — notifies engineers when competitor patents match predefined technology areas; (3) *Analysis agent* — generates competitive intelligence reports on demand.
- **Auth & security:** Full RBAC via AD groups. TLS everywhere. Audit log for all queries (who asked what, which patents returned). Data never leaves the on-prem perimeter.
- **Deployment:** **Kubernetes** (OpenShift or vanilla k8s). GitOps with ArgoCD. Canary deployments for LLM model updates.
- **Observability:** Prometheus + Grafana + Loki (logs) + Jaeger (tracing). SLA dashboards for business stakeholders.

#### Resiliency
- **Application layer:** 3+ pod replicas behind k8s Service + Ingress. Horizontal Pod Autoscaler.
- **LLM:** Active/active GPU nodes behind vLLM load balancer. Automatic node failover.
- **Qdrant:** 3-node cluster with replication factor 2. Node failure transparent to application.
- **Neo4j:** Causal cluster (3 core servers). Automatic leader election.
- **PostgreSQL (metadata):** Patroni-managed HA cluster with automatic failover.
- **Storage:** RAID-6 NAS with off-site backup replication.
- **DR:** Warm standby site. **RTO < 4 hours, RPO < 1 hour**.
- **Target availability:** **99.9% (planned maintenance excluded)**.

#### Scalability
- Kubernetes HPA scales stateless components (app pods, ingestion workers) automatically.
- vLLM batches inference requests — GPU utilisation > 80% typical.
- Qdrant horizontal sharding for corpus growth beyond 5M chunks.
- Airflow task queues absorb burst ingestion without data loss.

#### Costs
| Item | Estimate |
|---|---|
| GPU cluster (4× A100 80 GB server) | €200,000–350,000 one-time |
| k8s cluster servers (control plane + workers) | €40,000–80,000 one-time |
| Neo4j Enterprise licence | €25,000–50,000/year |
| NAS upgrade + DR storage | €20,000–40,000 one-time |
| Engineering (build + integration + testing) | €120,000–180,000 |
| Operations & maintenance (annual) | €60,000–100,000/year |
| **Total production build** | **€400,000–650,000** |
| **Annual running costs** | **€85,000–150,000/year** |

### Scenario B — AWS Cloud

#### Technology Stack
- **LLM:** Amazon Bedrock **Provisioned Throughput** for `claude-3-5-sonnet` (guaranteed tokens/second, predictable latency < 2 s). Provisioned capacity negotiated with AWS for committed usage discount (typically 20–40% savings vs. on-demand at scale).
- **Vector search:** **Amazon OpenSearch Serverless** (vector search collection). Scales automatically with zero capacity planning. Alternatively: **Pinecone** (managed, best-in-class ANN performance) via API.
- **Knowledge Graph:** **Amazon Neptune** (provisioned, 2-AZ). Supports both property graphs (Gremlin) and RDF (SPARQL). Existing NetworkX traversal logic maps cleanly onto Neptune queries.
- **Ingestion pipeline:** Fully event-driven: S3 upload → **EventBridge** rule → **Step Functions** state machine (extract → chunk → embed → index → notify). New patent searchable within 30 minutes of S3 upload. Parallel processing via Lambda + ECS Fargate workers.
- **Document management integration:** AWS native connectors for **SharePoint Online** (via Microsoft 365 API + Lambda), **S3-based DMS** (direct), or custom connectors for on-prem DMS via **AWS DataSync** + **Direct Connect / Site-to-Site VPN**.
- **Agentic AI:** **Amazon Bedrock Agents** with custom action groups. Three agents mirroring on-prem design: Ingestion, Alert, and Analysis. **Bedrock Knowledge Bases** as a managed alternative to the custom retrieval stack for simpler use cases. **Step Functions** for multi-step agent orchestration with audit trail.
- **Auth & security:** AWS Cognito + SAML federation to Ferrosyn IdP. **AWS IAM** for service-to-service. **KMS** for patent data encryption at rest. **AWS Macie** for sensitive data detection. **VPC endpoints** keep traffic off the public internet. **CloudTrail** for full audit log.
- **Deployment:** **EKS** (Kubernetes on AWS) for stateful workloads. ECS Fargate for stateless. **CodePipeline + CodeDeploy** for CI/CD. **AWS Config** for compliance monitoring.
- **Observability:** CloudWatch + X-Ray + OpenSearch Dashboards (operational metrics). **AWS Cost Explorer** for spend visibility.

#### Resiliency
- **Application (EKS/ECS):** Multi-AZ node groups, pod anti-affinity rules, ALB health checks.
- **Bedrock:** AWS-managed, 99.9% SLA, no infrastructure to manage.
- **OpenSearch Serverless:** Fully managed, multi-AZ, automatic failover.
- **Neptune:** 2-AZ with automatic failover < 30 s. Point-in-time restore up to 35 days.
- **Aurora PostgreSQL:** Multi-AZ, read replicas, Global Database for DR.
- **S3:** Cross-region replication to secondary AWS region for DR.
- **DR:** Active/passive cross-region. **RTO < 1 hour, RPO < 15 minutes**.
- **Target availability:** **99.95%** (composite of AWS managed service SLAs).

#### Scalability
- Bedrock Provisioned Throughput handles sustained load without cold starts.
- OpenSearch Serverless auto-scales OCU (compute units) with traffic.
- EKS Cluster Autoscaler adds/removes nodes based on pod demand.
- Step Functions + SQS absorb burst ingestion (1,000 patents/day achievable).
- Neptune auto-scales read replicas for high query volume.

| OpenSearch Serverless (4 OCU baseline) | €700–1,200 |

AWS has a lower barrier to entry, faster time-to-value, and lower operational overhead, but ongoing costs grow with usage. At Ferrosyn's anticipated query volumes, the break-even is approximately **Year 3–4**.

## Key Decision Factors

### Choose On-Premise if:
- Patent data is classified or subject to strict data residency requirements
- Ferrosyn has an existing IT operations team and hardware procurement process
- Long-term cost optimisation (5+ year horizon) is the priority
- Integration with on-prem systems (OpenText, SAP) requires low-latency internal network access
- Regulatory / audit requirements mandate full infrastructure control

### Choose AWS if:
- Speed to production is the primary driver
- IT team prefers managed services over infrastructure operations
- Query volumes are unpredictable (pay-per-use is more efficient)
- Ferrosyn already has AWS infrastructure, agreements, or an Enterprise Discount Programme
- Multi-region DR and 99.95%+ availability is a hard requirement

## Agentic AI in Production

Both deployment scenarios converge on the same three-agent architecture at production scale:

| Agent | Function | On-Premise | AWS |
|---|---|---|---|
| **Ingestion Agent** | Monitors patent sources (EPO, USPTO, internal DMS), classifies new filings, triggers processing pipeline | LangGraph + Airflow DAG | Bedrock Agent + Step Functions |
| **Alert Agent** | Proactively notifies engineers when new competitor patents match predefined technology profiles | LangGraph scheduled task | Bedrock Agent + EventBridge scheduler |
| **Analysis Agent** | Generates on-demand competitive intelligence reports, claim mapping, and prior-art summaries | LangGraph + custom tools | Bedrock Agent + Knowledge Bases |

## Recommended Path

1. **PoV on AWS** — lowest friction, fastest setup, validates the business case without capital commitment. Use Bedrock Claude Haiku for cost-controlled LLM calls during the pilot.
2. **MVP on AWS** — leverage managed services (OpenSearch, Bedrock, Fargate) to focus engineering effort on domain quality (retrieval tuning, entity extraction, evaluation) rather than infrastructure.
3. **Production decision at MVP exit** — after 3–6 months of production-like load data, make an informed build-vs-buy decision on on-prem GPU infrastructure vs. Bedrock Provisioned Throughput based on actual query volumes and data residency requirements.
